### 1.2 技术栈选型评估

| 技术组件 | 选型 | 评估 | 替代方案 |
|----------|------|------|----------|
| **AI平台** | 扣子（Coze） | ✅ 稳定，生态成熟 | Dify/Flowise |
| **大模型** | 默认GPT-4 | ⚠️ 成本高，需评估 | Claude/文心一言 |
| **数据存储** | 飞书多维表格 | ✅ 与交付包一致 | Airtable/Notion |
| **自动化** | 扣子工作流 | ⚠️ 功能有限 | Make/Zapier |
| **备份方案** | 无 | 🔴 风险高 | 需建立 |

**关键建议：**

1. **大模型成本优化**
   ```
   分层调用策略：
   - 人设提取：GPT-4（精度要求高）
   - 文案生成：GPT-3.5/Claude（成本优化）
   - 内容审核：文心一言（中文优化）
   
   预计成本降低：60%
   ```

2. **建立双平台备份**
   ```
   主平台：扣子（主要运行）
   备用平台：Dify（紧急切换）
   
   切换触发条件：
   - 扣子服务中断>2小时
   - API错误率>5%
   - 响应时间>10秒
   ```

## 二、关键技术卡点分析

### 2.1 卡点1：扣子系统配置效率（风险：🔴 高）

**问题描述：**
- 每单需2小时配置扣子系统
- 配置过程重复性高，人工操作多
- 扣子平台功能变更时，配置流程需重新学习

**技术解决方案：**

```
方案A：模板化配置（推荐）

1. 建立3套行业母版工作流
   ├── 专业人士版（律师/咨询师/疗愈师）
   ├── 门店版（咖啡/花店/餐饮）
   └── 通用版（其他行业）

2. 一键克隆机制
   - 母版工作流 → 克隆 → 自动替换客户变量
   - 变量包括：姓名/行业/人设标签/客单价等
   - 配置时间：2小时 → 30分钟

3. 自动化脚本（可选进阶）
   - 扣子API调用（如开放）
   - 自动创建工作流+替换变量
   - 配置时间：30分钟 → 5分钟
```

**实施优先级：** 🔴 P0（启动前必须完成）

---

### 2.2 卡点2：信息采集自动化（风险：🟡 中）

**问题描述：**
- 28个问题需人工整理，效率低
- 信息质量不稳定，影响后续交付
- Day 2追问补充依赖老徐时间

**技术解决方案：**

```
信息采集自动化流程：

1. 飞书表单自动化
   - 客户付款后 → 自动发送飞书表单
   - 表单包含：选择题70% + 填空题30%
   - 支持语音上传（关键问题）

2. AI自动整理
   - 表单提交后 → 自动触发扣子工作流
   - AI提取：人设标签/关键词/金句
   - 生成：信息采集报告（结构化）

3. 质量评分自动化
   - 自动评分：信息完整度/质量
   - 低于3分 → 自动提醒客户补充
   - 高于3分 → 直接进入Day 2

4. 老徐审核环节
   - 仅需审核AI生成的信息采集报告
   - 时间：1小时 → 15分钟
```

**技术实现：**
- 飞书表单 → Webhook → 扣子工作流
- 预计开发时间：2天
- 预计节省时间：每单45分钟

**实施优先级：** 🟡 P1（第5单前完成）

### 2.3 卡点3：内容生成质量稳定性（风险：🟡 中）

**问题描述：**
- AI生成文案风格不稳定
- 有时出现"AI味"，不符合老徐品牌调性
- 需要人工审核和修改

**技术解决方案：**

```
方案：品牌调性固化机制

1. 建立品牌调性知识库
   ┌─────────────────────────────────────┐
   │ 老徐品牌调性知识库                   │
   ├─────────────────────────────────────┤
   │ 语言风格：                           │
   │ - 短句优先，每句不超过15字           │
   │ - 少用形容词，多用动词               │
   │ - 避免"干货""福利"等电商词          │
   │                                     │
   │ 禁忌句式：                           │
   │ - ❌ "XX才是真正的..."              │
   │ - ❌ "你值得更好的..."              │
   │ - ❌ "我想和你聊一聊..."            │
   │                                     │
   │ 常用结构：                           │
   │ - 痛点+解决方案+行动指令             │
   │ - 案例+数据+对比                     │
   │ - 反常识观点+解释+佐证               │
   └─────────────────────────────────────┘

2. Prompt工程优化
   - 系统Prompt中固化品牌调性
   - Few-shot示例（每条带3-5个示例）
   - 输出格式标准化

3. 质量检测自动化
   - 自动检测：禁忌词/句式长度/形容词密度
   - 检测不通过 → 自动重生成
   - 检测通过 → 进入人工审核

4. 反馈学习机制
   - 客户选择"优质文案" → 加入Few-shot库
   - 客户修改文案 → 记录修改点，优化Prompt
   - 每月更新品牌调性知识库
```

**实施优先级：** 🟡 P1（第10单前完成）

### 2.4 卡点4：数据追踪与回流（风险：🟢 低）

**问题描述：**
- 客户发布后的效果数据无法自动回收
- 无法基于数据优化文案风格
- 缺少A/B测试能力

**技术解决方案：**

```
数据追踪方案（轻量版）：

1. 发布记录追踪
   - 客户在飞书表格手动记录：发布内容/时间
   - 每周客户截图：互动数据（点赞/评论/私信）
   - 老徐助理每周汇总

2. 数据追踪方案（自动化进阶版）：
   - 客户授权朋友圈数据读取（技术难度高，暂不建议）
   - 替代方案：客户每日在系统打卡，AI自动分析

3. 效果反馈闭环
   - 高互动文案 → 标记为"优质" → 加入模板库
   - 低互动文案 → 分析原因 → 优化生成策略
   - 每月生成：客户内容效果报告
```

**实施优先级：** 🟢 P2（第30单后考虑）

## 三、系统稳定性保障方案

### 3.1 扣子平台风险应对

| 风险场景 | 概率 | 影响 | 应对措施 |
|----------|------|------|----------|
| 服务中断 | 中 | 高 | 备用平台+人工兜底 |
| API限流 | 中 | 中 | 多账号轮换+队列机制 |
| 功能变更 | 高 | 中 | 版本锁定+监控告警 |
| 数据丢失 | 低 | 高 | 定期备份+导出 |

**应急预案：**

```
预案A：扣子服务中断

触发条件：服务中断>2小时 或 API错误率>10%

应急措施：
1. 立即启动备用平台（Dify）
2. 通知所有客户切换至备用系统
3. 老徐启动人工兜底（手写文案）
4. 48小时内修复或迁移

客户沟通：
"系统临时升级，已切换至备用通道。
这7天的内容我亲自帮你写，额外送你30天陪跑。"
```

```
预案B：API限流

触发条件：单账号API调用>1000次/小时

应急措施：
1. 启用多账号轮换机制（准备3个扣子账号）
2. 建立请求队列，错峰处理
3. 非紧急请求延迟处理

预防措施：
- 预估峰值：100客户 × 10次/天 = 1000次/天
- 单账号容量：1000次/小时 × 24 = 24000次/天
- 结论：单账号可支撑，但需准备备用账号
```

**数据分类与保护：**

| 数据类型 | 存储位置 | 备份频率 | 保留期限 |
|----------|----------|----------|----------|
| 客户信息采集 | 飞书文档 | 实时 | 3年 |
| 扣子工作流配置 | 扣子平台 | 每周导出 | 永久 |
| 生成文案记录 | 飞书表格 | 每日 | 1年 |
| 客户使用数据 | 飞书表格 | 每周 | 1年 |

**备份机制：**

```
自动化备份流程：

1. 扣子工作流导出
   - 每周日自动导出所有工作流配置
   - 存储位置：飞书云盘/备份文件夹
   - 保留：最近10个版本

2. 客户数据备份
   - 每日自动备份飞书表格
   - 导出格式：Excel + CSV
   - 存储位置：本地NAS + 云盘双备份

3. 灾难恢复
   - 主数据丢失 → 从备份恢复（RTO<4小时）
   - 扣子平台故障 → 切换备用平台（RTO<2小时）
```
